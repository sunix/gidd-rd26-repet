GitHub Issue Driven Development

  • 1️⃣ Créer une GitHub Issue (depuis VS Code ou le téléphone)
  • 2️⃣ Assigner à GitHub Copilot (Coding Agent)
  • 3️⃣ Suivre la session live — voir ce qu'il fait
  • 4️⃣ Revue de code de la PR générée
  • 5️⃣ Commenter / demander des corrections
  • 6️⃣ Merger la PR ✅
100% cloud, sans IDE Depuis le téléphone Comme un mainteneur OSS
Synthèse de mon flow : je crée une issue, je l'assigne à Copilot et je fais des revues de code. Avantages : voir la session live pour voir ce qu'il fait, sans IDE, 100% dans le cloud, la PR générée comme si on était mainteneur d'un projet open source et qu'on faisait la différence, et les screenshots avec un serveur MCP Playwright. Ça n'avait pas marché du premier coup, en tout cas pour certaines PR (ex. gift-card #3).

Créer une issue depuis VS Code

Pour que ça marche : attention à l'environnement de preview pour les PR (surge.sh), au firewall, et à agent.md qui évite de refaire une analyse déjà connue du projet. On peut créer des issues directement dans VS Code, en mode conversation.

Exemple : jdtls-mcp. Non, il n'y arrive pas, forcément — projet trop complexe. Solution : petites itérations, une bonne pratique de dev classique — découper en petites tâches.

Solution : petites itérations / MVP 🛹→🚗

  • Documenter les prochaines étapes dans agent.md / README
  • Comme en vrai développement : pas de PR trop grosse !
Documenter les prochaines étapes (java-lsp-mcp-server README "Next Steps"), avec des issues de suivi. Comme dans les vrais projets, on ne veut pas de PR trop grosse : les tâches complexes se divisent en sous-étapes. Quand il y a du flou ou besoin d'expérimentation, passer par ChatGPT / Copilot Chat d'abord, puis guider l'agent étape par étape.

Ce qui marche moins bien

  • Interactions git complexes (rebase, push force)
  • Exécution locale (ex: tester des rendus kdenlive)
  • Quand y'a besoin d'expérimentation → ChatGPT / Copilot Chat d'abord
💡 Ne pas laisser l'IA seule !
Demandez-lui de générer des scripts Playwright pour retester à l'identique dans la CI
Ce qui marche moins bien : les interactions avec git (ex. rebase pour récupérer les derniers changements, car l'agent ne peut pas faire de push force), et quand il y a besoin d'exécuter des choses localement (ex. tester des rendus kdenlive, cf. Tennis Match Scoring Web App). Ne pas laisser l'IA juste tester votre application : demandez-lui de générer un script (ex. Playwright) qui retestera à l'identique votre app, et d'affiner avec le temps (idem pour les releases). Dans un agent.md, dites-lui de documenter un maximum pour ne pas refaire la même analyse, et de le mettre à jour quand il le faut. Donnez des definitions of done précises, coupez en petites étapes pour qu'il puisse itérer et récupérer votre feedback.

Mon gros frein a été le changement des tarifs de GitHub Copilot (passage au usage-based billing) : en 5 jours, le 5 juin, j'avais consommé 50% de mes crédits. Je suis passé à Claude Code (forfait Claude Code Team pris par Sciam). Claude fait bien le boulot, mais j'étais tout le temps scotché sur mon PC. Claude Code est moins intuitif : Copilot utilise certains MCP comme Playwright plus facilement. Surtout, fini les interactions avec l'agent sur les issues et PR GitHub : tout se fait en local sur un PC avec un IDE. En terme de charge mentale, le GitHub Issue Driven Development est moins intrusif : je pouvais passer 1 min sur mon téléphone pendant que ma fille était à la piscine pour commenter une PR, alors qu'avec Claude Code je dois rester sur mon PC. Néanmoins, j'ai fini par installer récemment Claude GitHub Action sur certains projets, ce qui me permet un flow similaire à GIDD (ex. tc11.fr, en annotant @Claude dans une issue).

Conclusion

Suivez les bonnes pratiques de dev ! 🏆

Mettez-vous à la place de l'IA en tant que personne
  • 📝 Documentation — elle s'en sert comme un humain
  • 📋 Spec claire — définitions of done précises
  • 🔧 Choix techniques documentés dans agent.md
  • 🎯 Petites tâches — ne pas lui demander trop d'un coup
  • 🔄 Itérations — votre feedback est précieux
Oui, au final, suivez les bonnes pratiques de dev ! Si vous suivez les bonnes pratiques que l'on connaît tous, l'IA va savoir mieux travailler. Mettez-vous à la place de l'IA en tant que personne (documentation, spec, choix technique, règles à suivre, definition of done) : l'agent va en tenir compte comme un humain, voire mieux. Ça me permet de bosser sur des sujets, de garder le rythme.
