Chapitre 5

Mon flow GIDD — la synthèse

GitHub Issue Driven Development

  • 1️⃣ Créer une GitHub Issue (depuis VS Code ou le téléphone)
  • 2️⃣ Assigner à GitHub Copilot (Coding Agent)
  • 3️⃣ Suivre la session live — voir ce qu'il fait
  • 4️⃣ Revue de code de la PR générée
  • 5️⃣ Commenter / demander des corrections
  • 6️⃣ Merger la PR ✅
100% cloud, sans IDE Depuis le téléphone Comme un mainteneur OSS
Avantages : voir la session live, 100% dans le cloud, les screenshots avec MCP Playwright. Comme si on était mainteneur d'un projet open source.

Chapitre 6

Réfutations & Bonnes pratiques

Attention au spam ! 📬

💡 Solution : travailler dans un fork, soumettre la PR quand c'est prêt

Ça ne marche pas pour les projets complexes

  • Exemple : jdtls-mcp — trop complexe d'un coup
  • Solution : petites itérations
  • Documenter les prochaines étapes dans agent.md / README
  • Comme en vrai développement : pas de PR trop grosse !
↗ github.com/sunix/java-lsp-mcp-server

Ce qui marche moins bien

  • Interactions git complexes (rebase, push force)
  • Exécution locale (ex: tester des rendus kdenlive)
  • Quand y'a besoin d'expérimentation → ChatGPT / Copilot Chat d'abord
💡 Ne pas laisser l'IA seule !
Demandez-lui de générer des scripts Playwright pour retester à l'identique dans la CI
